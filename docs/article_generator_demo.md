# 铸闻 — 演示案例（3 个真实场景）

> 说明：以下案例依据本系统的**真实数据流**还原——即「原始证据 → Fact Sheet → 新闻初稿 → 逐句核查 → 待核实 → 人工修改 → 导出」的完整链路。示例数据用于演示管线行为（在无 LLM Key 时由**规则回退**生成，有 Key 时由真实模型生成，二者管线一致、Fact Sheet 与核查逻辑相同）。

---

## 案例一：投融资新闻（financing_news）

**① 原始证据**（选题卡关联的 articles / intelligence 摘要）
```
[E0] 来源=XX财经｜2026-03-12｜标题=「云擎科技完成 B 轮 3 亿元融资，启明创投领投」
     摘要=云擎科技今日宣布完成 B 轮融资，金额约 3 亿元，由启明创投领投，源码资本跟投。
[E1] 来源=创业邦｜2026-03-12｜标题=「云擎科技 B 轮融资落定，资金将用于 AI 算力研发」
     摘要=本轮融资估值较上轮提升明显，资金用途主要投向 AI 算力基础设施。
```

**② Fact Sheet（系统自动提取）**
```json
{
  "subject": "云擎科技",
  "eventType": "投融资",
  "eventDate": "2026-03-12",
  "location": "",
  "confirmedFacts": [
    {"field":"主体","value":"云擎科技","sourceIds":["E0","E1"]},
    {"field":"金额","value":"3 亿元","sourceIds":["E0"]},
    {"field":"轮次","value":"B 轮","sourceIds":["E0","E1"]},
    {"field":"投资方","value":"启明创投","sourceIds":["E0"]}
  ],
  "conflictingFacts": [],
  "unverifiedFacts": [
    {"field":"资金用途","value":null,"sourceIds":[]},
    {"field":"公司主营业务","value":null,"sourceIds":[]}
  ],
  "entities": ["云擎科技","启明创投","源码资本"],
  "fields": {"融资主体":"云擎科技","融资轮次":"B 轮","融资金额":"3 亿元","投资机构":"启明创投","融资时间":"2026-03-12","资金用途":null,"公司主营业务":null,"历史融资":null}
}
```

**③ 新闻初稿（节选）**
> **标题**：云擎科技完成 B 轮融资，金额约 3 亿元
> **导语**：云擎科技近期完成 B 轮融资，涉及金额约 3 亿元，本稿依据已抓取证据整理，发布前须人工审核。
> **融资事件**：云擎科技完成 B 轮融资，2026-03-12 相关消息发布。本轮由启明创投参与。
> **融资金额与轮次**：据证据，融资金额约为 3 亿元。轮次为 B 轮。
> **资金用途**：资金用途未在证据中明确披露，需向企业核实。

**④ 逐句核查（节选）**
| 句 | 状态 | 说明 |
|---|---|---|
| 云擎科技完成 B 轮融资… | 🟢 supported | 与 E0/E1 轮次、主体可直接对应 |
| 融资金额约为 3 亿元 | 🟢 supported | 与 E0 金额 token 对应 |
| 资金用途未在证据中明确披露 | 🔵 background | 背景性说明 |

整体状态：**partial**（存在未确认字段，但无「未证实」断言）。

**⑤ 待核实事项（系统生成）**
- 资金用途未在证据中确认，需向企业核实。
- 公司主营业务未在证据中详述，建议补充公司简介。

**⑥ 人工修改**：编辑补全「资金用途：投向 AI 算力基础设施（据 E1）」，并将 `status` 置 `published`。

**⑦ 导出**：`GET /api/generated-articles/[id]/export?format=markdown` 得到含来源、待核实、风险提示的 Markdown 稿件。

## 案例二：公司动态（company_news）

**① 原始证据**
```
[E0] 来源=证券时报｜2026-04-03｜标题=「广汽部件与华为达成智能驾驶合作」
     摘要=广汽部件宣布与华为在智能驾驶域控制器方面达成合作，将于年内量产。
```

**② Fact Sheet（节选）**
```json
{
  "subject": "广汽部件",
  "eventType": "公司业务动态",
  "confirmedFacts": [
    {"field":"主体","value":"广汽部件","sourceIds":["E0"]},
    {"field":"投资方","value":"华为","sourceIds":["E0"]}
  ],
  "fields": {"公司主体":"广汽部件","业务动作":null,"合作对象":"华为","发生时间":"","涉及地区":null,"战略目的":null}
}
```

**③ 新闻初稿（节选）**
> **导语**：广汽部件近期有业务动态，本稿依据已抓取证据整理，发布前须人工审核。
> **业务动作**：广汽部件近期有业务动态。
> **产品或合作内容**：具体产品、合作内容未在证据中详述，需核实。
> **公司背景**：广汽部件为本次动态的主体，主营业务与背景建议补充。

**④ 逐句核查**
- 「广汽部件近期有业务动态」🔵 background（无具体事实 token）
- 「具体产品…需核实」🔵 background
- 单一来源：系统风险提示「目前仅单一来源，建议补充权威信源后再发布」

整体状态：**partial**（单一来源 + 待核实字段）。

**⑤ 待核实事项**
- 相关事实无法从现有证据确认，需补充核实。（单一来源提示已追加）

**⑥ 人工修改**：编辑补充合作细节与战略意义，并补充第二来源后发布。

## 案例三：并购新闻（ma_news）

**① 原始证据**
```
[E0] 来源=A媒体｜2026-05-09｜标题=「中科收购蓝天环保 51% 股权，交易额 12 亿元」
     摘要=中科集团拟以 12 亿元收购蓝天环保 51% 股权，构成控股收购。
[E1] 来源=B媒体｜2026-05-09｜标题=「中科集团收购蓝天环保，交易额传为 12 亿」
     摘要=多家媒体称交易额约 12 亿元，但交易对方尚未官宣。
```

**② Fact Sheet（冲突检测）**
```json
{
  "subject": "中科集团",
  "confirmedFacts": [
    {"field":"主体","value":"中科集团","sourceIds":["E0","E1"]},
    {"field":"金额","value":"12 亿元","sourceIds":["E0","E1"]},
    {"field":"股权比例","value":"51%","sourceIds":["E0"]}
  ],
  "conflictingFacts": [
    {"field":"被收购方","value":"蓝天环保 / 交易对方尚未官宣","sourceIds":["E0","E1"]}
  ],
  "fields": {"收购方":"中科集团","被收购方":null,"交易金额":"12 亿元","股权比例":"51%","交易方式":null,"交易目的":null,"审批状态":null,"交割状态":null}
}
```
> 冲突检测发现「被收购方」说法不一（E0 明确「蓝天环保」、E1「交易对方尚未官宣」），归入 `conflictingFacts`，**绝不写入确定性正文**，仅在待核实中提示。

**③ 新闻初稿（节选）**
> **交易概况**：中科集团近期涉及并购交易，交易金额约为 12 亿元。
> **交易双方**：中科集团为本次交易相关方。涉及股权比例约 51%。（被收购方存在来源冲突，已在待核实标注）
> **金额与股权比例**：交易金额 12 亿元，股权比例 51%。
> **审批与交割**：审批状态与交割进度需以监管公告为准。

**④ 逐句核查**
- 「交易金额约为 12 亿元」🟢 supported（E0/E1 金额对应）
- 「涉及股权比例约 51%」🟢 supported（E0 股权对应）
- 「审批状态与交割进度需以监管公告为准」🔵 background

整体状态：**partial**。

**⑤ 待核实事项（系统生成）**
- 被收购方存在多方不一致（蓝天环保 / 交易对方尚未官宣），请人工核实后选用。
- 部分关键事实存在来源冲突，发布前须交叉核实。

**⑥ 人工修改**：编辑据 E0 官方口径确认「蓝天环保」，删除冲突提示后发布。

## 三案例对照

| 维度 | 投融资 | 公司动态 | 并购 |
|---|---|---|---|
| 类型 | financing_news | company_news | ma_news |
| 证据数 | 2（一致） | 1（单一来源） | 2（存在冲突） |
| 冲突检测 | 无冲突 | 单一来源提示 | 被收购方冲突→待核实 |

> 三个案例完整覆盖了 §16 要求的「投融资 / 公司动态 / 并购（或中标）」三类，并均展示了 原始证据 → Fact Sheet → 初稿 → 逐句核查 → 待核实 → 人工修改 → 导出 的全链路。

## 案例四：铸闻工作台操作流（推荐选题 → 新建内容 → 微信发布）

> 演示「不靠单卡逐个点」的批量作业方式。以下为铸闻工作台 `/generated-articles` 的真实交互链路。

### ① 今日推荐选题（自动筛选题）

打开工作台，系统调用 `GET /api/generated-articles/recommendations?limit=8`，按
`total = card.totalScore*0.6 + 来源多样性*0.2 + 新鲜度*0.2` 自动排出最值得写的 8 张卡：

```
[推荐#1] 云擎科技 B 轮融资（total 0.82，suggestedType=financing_news，尚未生成）
[推荐#2] 中科收购蓝天环保（total 0.77，suggestedType=ma_news，已有稿件 ga-xxxx）
[推荐#3] 广汽部件×华为合作（total 0.71，suggestedType=company_news，尚未生成）
……
```

编辑勾选 #1、#3 后点「批量生成」→ `POST /api/generated-articles/batch-generate`
（顺序生成、上限 10 篇）→ 返回 `{ okCount:2, failCount:0, results:[...] }`。

### ② 新建内容（无选题卡，库内检索 或 手填证据，或并用）

点「＋ 新建内容」，`NewArticleDialog` 流程：

1. 输入主题：`请用库内证据写一篇关于「某新能源公司中标国网项目」的报道`；
2. 对话框自动 `GET /api/evidence/search?q=新能源 中标 国网` 拉出相关 evidence，默认勾选前 8 条，可手动增删；
3. **也可完全不依赖库内数据**：展开「手动添加证据」粘贴手上材料，例如
   ```
   标题：某新能源公司中标国网 5G 基站储能项目
   内容：中标金额约 2.3 亿元，工期 18 个月，为其年内最大订单
   来源：招标文件（手填）
   日期：2026-07-18
   ```
   该条以 `inline-0` 合成 id 进入管线，内容冗余落库，编辑器「证据」面板照常显示；
4. 选类型/风格/长度后提交 `POST /api/generated-articles/custom`（库内勾选项随 `evidenceIds`、手填项随 `evidenceItems` 一并发送）；
5. 系统合成 `custom-<ts>` 选题卡 → 走与单卡完全相同的 Fact Sheet→生成→核查→落库管线；跳转编辑器，稿件标记为「自建」，无「选题卡」回链。

> 即：库内有料就搜库、库里没有就手填，**两条路都走同一套 Fact Sheet→生成→核查管线**，保证有据可查。

### ③ 生成前勾选数据来源

在选题卡页「一键生成」弹窗（`GenerateArticleDialog`）中，挂载即拉取该卡证据并呈现可勾选面板：

```
数据来源（3/5 已勾选）  [全选] [清空]
 ☑ 来源A｜2026-07-01｜标题…
 ☐ 来源B｜2026-06-28｜标题…（不相关，取消）
 ☑ 来源C｜2026-07-03｜标题…
 ☑ 来源D｜2026-06-30｜标题…
 ☐ 来源E｜2026-05-12｜标题…（过期，取消）
```

仅勾选项随 `evidenceIds` 传入生成管线，未勾选的被 service 层过滤，避免污染初稿。

### ④ 发布到微信公众号

在编辑器右上「发布到公众号 ▾」(`WechatPublishMenu`)：

- **方式一 · 复制排版（零配置）**：取 `GET /export?format=wechat` 返回的 `{ title, digest, html }`，
  将内联样式 HTML 写入剪贴板（text/html + text/plain）并打开 `mp.weixin.qq.com`，直接粘贴。
- **方式二 · API 推送（需凭证）**：若已配 `WECHAT_APPID`/`WECHAT_APPSECRET`/`WECHAT_THUMB_MEDIA_ID`，
  点「推送草稿」→ `POST /publish-wechat` 获取 `access_token` → 推送为草稿，返回 `{ ok:true, mediaId }`；
  未配凭证时返回 `501` 并提示改用复制方式。

## 案例五：生成大盘综述（免费源下载大盘 → 铸闻）

把"调用免费行情源下载整个股市数据再写大盘综述"落成一键操作，且**复用与「新建内容」完全相同的管线**——符合「自动生成概括整个股市的财经新闻、讲指数与涨跌家数」的产品定位。

**操作**
1. 铸闻工作台点「生成大盘综述」，`MarketNewsDialog` 预览大盘快照（无需输入任何个股代码）。
2. 点「预览」→ `GET /api/generated-articles/market-overview` 返回：`日期 / 时间 / 主题（A股大盘综述（YYYY-MM-DD）） / 5 大指数（点位·涨跌点·涨跌幅） / 涨跌家数（上涨 X 家·下跌 Y 家·平盘 Z 家）`，确认无误。
3. 选「财经媒体体 / 600 字」→ 点「生成大盘综述」→ `POST /api/generated-articles/market-overview`。

**后端链路（自动）**
- `getMarketSnapshot()` 并行拉取：新浪简要指数行情（`getMarketIndices`，GBK 解码，5 大指数 `DEFAULT_INDEX_CODES` = 上证指数/深证成指/创业板指/沪深300/科创50）+ 东方财富涨跌家数（`getMarketBreadth`）。
- `buildMarketTopic(date)` 派生主题「A股大盘综述（日期）」；`buildMarketEvidence(snap)` 把快照转成两条「库外证据」：
  - ① 指数快照：各指数点位 / 涨跌点 / 涨跌幅 / 量额 / 行情时间。
  - ② 涨跌家数：上涨 / 下跌 / 平盘家数（东方财富不可达时 `available:false` → 自动降级为「涨跌家数暂不可用」提示证据）。
- 直接调用 `generateCustomArticle({ topic, evidenceItems, options:{articleType:'market_overview', ...} })` → Fact Sheet → 生成 → 逐句核查 → 落库（合成 `custom-<hash>` 选题卡）。编辑器「证据」面板可看到两条大盘证据及其来源（新浪财经指数 / 东方财富）。

**市场覆盖**
- 全部 A股 大盘维度，无任何个股代码输入——指数与涨跌家数均来自免费源（无需 Key）。
- 东方财富涨跌家数在免费源限流时返回 `available:false`，后端自动降级，稿件仍基于指数快照生成。
